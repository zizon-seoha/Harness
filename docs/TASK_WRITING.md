# Task Writing

이 문서는 task 문서를 작성하거나 구현 후 갱신할 때 따라야 하는 규칙입니다.

## 기본 원칙

- task 문서는 현재 단계에서 필요한 결과와 검증 기준을 고정하는 문서입니다.
- task 문서에는 기본적으로 What, 즉 무엇이 필요한지를 적습니다.
- 구현 전 task 문서에는 How, 즉 클래스 구조, 호출 순서, 내부 구현 방식, Inspector 연결 방법을 추측해서 적지 않습니다.
- 구현 후에는 실제 추가된 코드, 컴포넌트명, Inspector 필드명, 필요한 씬 오브젝트 기준으로 task 문서를 갱신합니다.
- 사용자 요청과 현재 task 범위를 벗어나는 항목은 task 문서에 추가하지 않습니다.
- 다음 task에서 구현할 흐름이나 아직 연결되지 않은 자동 동작을 현재 task의 검증 항목처럼 적지 않습니다.
- 새 task 문서를 생성하면 `docs/PLANS.md` 작업리스트의 해당 단계 `Task 문서` 칸도 생성한 task 문서 링크로 갱신합니다.

## 작성 시점별 규칙

### 구현 전

- `구현 항목`에는 현재 task에서 달성해야 하는 결과만 적습니다.
- `범위 밖`에는 이번 task에서 일부러 다루지 않을 항목을 적습니다.
- `사전 전제`에는 이전 단계 산출물이 필요할 때만 실제 전제 대상을 적습니다.
- `Unity Editor 작업 (수동)`과 `사람 확인`은 실제 코드가 없으면 placeholder 수준으로 둡니다.
- 실제 이름을 알 수 없는 오브젝트, 컴포넌트, 필드, 프리팹 이름을 추측해서 구체화하지 않습니다.

### 구현 후

- 완료된 `구현 항목`과 `Codex CLI 확인` 체크박스를 실제 완료 범위만 `[x]`로 갱신합니다.
- `Unity Editor 작업 (수동)`은 실제 추가된 컴포넌트와 Inspector 필드를 기준으로 구체화합니다.
- `사람 확인`은 실제 Play Mode에서 확인 가능한 런타임 흐름만 기준으로 구체화합니다.
- 사용자가 Unity Editor 수동 작업 또는 사람 확인 완료를 응답하기 전에는 해당 확인 항목을 완료 처리하지 않습니다.
- 사람 확인이 끝나기 전에는 다음 task 진행을 전제로 하는 문구를 쓰지 않습니다.
- 진행 중인 task에서 추가 변경이 발생해도 기존에 실제 완료된 항목은 유지하고, 새 변경으로 인해 새로 수행하거나 재확인해야 하는 항목만 `[ ]`로 둡니다.

## Unity Editor 작업 (수동) 작성 규칙

`Unity Editor 작업 (수동)`에는 Editor 안에서 사용자가 설정해야 하는 절차만 적습니다.

포함할 내용:

- Hierarchy에서 생성하거나 선택할 GameObject
- 부모/자식 오브젝트 구조
- Project 창에서 만들거나 저장할 Prefab
- 추가할 컴포넌트
- Inspector에서 설정할 필드와 값
- 어떤 오브젝트나 프리팹을 어떤 필드에 드래그해야 하는지
- 적용 후 정상 상태

포함하지 않을 내용:

- Play Mode에서 런타임 동작을 관찰하는 절차
- 다음 task에서 구현될 흐름
- 아직 구현되지 않은 자동 연결
- “연결 확인”, “설정 확인”처럼 대상과 방법이 없는 추상 문장

각 항목은 가능하면 아래 순서가 드러나게 작성합니다.

1. Hierarchy 또는 Project에서 무엇을 생성하거나 선택한다.
2. Inspector에서 어떤 컴포넌트를 추가하거나 확인한다.
3. 어떤 필드에 어떤 값 또는 참조를 넣는다.
4. 적용 후 무엇이 정상 상태인지 적는다.

### 나쁜 예

- `프리팹을 만든다.`
- `필드를 연결한다.`

### 좋은 예

- `Hierarchy에서 빈 GameObject를 만들고 이름을 Cargo_Box로 변경한다.`
- `Cargo_Box 아래에 Cube 자식 오브젝트를 만들고 표시용 Mesh로 사용한다.`
- `Cargo_Box 루트 오브젝트 Inspector에서 Add Component로 CargoItem을 추가한다.`
- `Project 창의 Assets/Prefabs/Cargo 폴더로 Cargo_Box를 드래그해 Prefab으로 저장한다.`
- `ConveyorCargoSpawner의 Cargo Prefabs 배열 Element에 Cargo_Box Prefab을 드래그하고 Type 값을 Box로 설정한다.`

## 사람 확인 작성 규칙

`사람 확인`에는 Play Mode에서 사람이 에디터 화면과 Inspector 상태를 보며 자연스럽게 관찰할 수 있는 런타임 동작만 적습니다.

포함할 내용:

- Unity Editor 컴파일 오류 없음 확인
- Play Mode 진입
- Play Mode에서 어디를 봐야 하는지
- 어떤 순서로 동작해야 정상인지
- 실패 시 Console 오류와 실제 관찰된 상태

포함하지 않을 내용:

- Play Mode 진입 전 Inspector 필드 연결 확인
- 컴포넌트 존재 여부 확인
- Inspector 값 설정 확인
- Inspector 필드 연결 방법
- Prefab 생성 절차
- 구현되지 않은 다음 단계 동작
- 현재 task 범위 밖의 자동화 흐름

각 항목은 가능하면 아래 순서가 드러나게 작성합니다.

1. `사람 확인` 모드인 경우 Unity Editor 컴파일 오류 없음 확인을 요청한다.
2. Play Mode에 진입한다.
3. 화면 또는 Inspector에서 자연스럽게 관찰할 대상을 확인한다.
4. 정상 동작 순서를 확인한다.
5. 실패 시 Console 오류와 실제 관찰된 상태를 기록한다.

### 나쁜 예

- `정상 동작을 확인한다.`
- `Play Mode에서 잘 되는지 본다.`
- `Play Mode 진입 전 필드가 연결되어 있는지 확인한다.`
- `Inspector 값을 확인한다.`

### 좋은 예

- `Play Mode 진입 후 Spawn Interval 값만큼 기다렸을 때 Spawn Point 위치에 CargoItem이 붙은 물류가 생성되는지 확인한다.`
