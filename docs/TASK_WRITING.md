# Task Writing

task 문서를 작성하거나 구현 후 갱신할 때 따르는 규칙. 새 task 문서는 `/new-task`로 생성한다.

## 기본 원칙

- task 문서는 현재 단계에서 필요한 결과와 검증 기준을 고정하는 문서다.
- task 문서에는 **What**(무엇이 필요한지)만 적는다.
- 구현 전에는 **How**(클래스 구조, 호출 순서, 내부 구현 방식)를 추측해서 적지 않는다.
- 구현 후에는 실제 추가된 코드(모듈명, 함수명, 엔드포인트, 잡 이름) 기준으로 갱신한다.
- 사용자 요청과 현재 task 범위를 벗어나는 항목은 추가하지 않는다.
- 다음 task에서 구현할 흐름이나 아직 연결되지 않은 동작을 현재 task의 검증 항목처럼 적지 않는다.
- 새 task 문서를 만들면 `docs/PLANS.md` 보드에 해당 행을 추가한다.

## 작성 시점별 규칙

### 구현 전

- `구현 항목`에는 이번 task에서 달성할 결과(What)만 적는다.
- `범위 밖`에는 이번 task에서 일부러 다루지 않을 항목을 적는다.
- `사전 전제`에는 이전 단계 산출물이 필요할 때만 실제 대상을 적는다.
- `사람 확인`은 실제 구현이 없으면 placeholder로 둔다. 이름을 모르는 모듈·엔드포인트를 추측해 구체화하지 않는다.

### 구현 후

- 완료된 `구현 항목`과 검증 항목 체크박스를 실제 완료 범위만 `[x]`로 갱신한다.
- `사람 확인`은 실제 런타임에서 확인 가능한 항목만 기준으로 구체화한다.
- 사용자가 사람 확인 완료를 응답하기 전에는 해당 항목을 완료 처리하지 않는다.
- 추가 변경이 생겨도 이미 완료된 항목은 유지하고, 새로 수행·재확인할 항목만 `[ ]`로 둔다.

## What / How 분리 — 나쁜 예 / 좋은 예

### 구현 항목 (What만)

나쁜 예 (How를 추측):

- `JobService 클래스에 run_at 필드를 추가하고 scheduler.add_job을 호출한다.`
- `useEffect로 fetch해서 상태에 저장한다.`

좋은 예 (What):

- `지정한 시각에 한 번 실행되는 예약 잡을 등록할 수 있다.`
- `대시보드 진입 시 최근 주문 목록이 표시된다.`

### 사람 확인 (Claude가 검증 못 하는 런타임·통합만)

`docs/STACK.md`의 「도메인 / 사람 확인 항목」을 인용한다. 자동 명령(ruff/pytest/tsc/vitest)으로 검증 가능한 것은 적지 않는다.

나쁜 예:

- `정상 동작을 확인한다.` (대상·방법 없음)
- `타입 오류 없는지 확인한다.` (자동 명령으로 검증 가능 → 사람 확인 아님)

좋은 예:

- `등록한 잡이 지정한 시각(예: 매일 09:00)에 실제로 1회 실행되는지 다음 날 로그로 확인한다.`
- `페이퍼트레이딩 모드에서 주문이 실거래 API로 전송되지 않는지 외부 콘솔에서 확인한다.`
