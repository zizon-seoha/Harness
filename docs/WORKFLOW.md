# Workflow

## 문서 우선순위

문서 해석 우선순위는 아래와 같습니다.

1. 사용자 직접 요청
2. `AGENTS.md`
3. 현재 진행 중인 task 문서
4. `docs/PLANS.md`
5. 나머지 프로젝트 문서

문서끼리 충돌하거나 현재 저장소 상태와 맞지 않으면, 구현 전에 불일치를 먼저 보고하고 수정 범위를 제안합니다.

## 작업 흐름

1. 관련 task와 선행 산출물을 확인합니다.
2. `docs/PLANS.md`에서 현재 단계 상태를 확인하고 작업 시작 시 `진행중`으로 갱신합니다.
3. 관련 코드와 참조 구조를 읽습니다.
4. 필요한 코드 변경을 구현합니다.
5. task 문서의 완료된 항목을 실제 범위만 반영합니다.
6. 구현 후 `docs/TASK_WRITING.md`의 작성 규칙에 맞춰 task 문서를 실제 코드 기준으로 갱신합니다.
   - `Unity Editor 작업 (수동)`과 `사람 확인` 항목은 갱신 직후 `docs/TASK_WRITING.md`의 해당 작성 규칙 및 나쁜 예/좋은 예와 대조해 규칙이 어긋난 부분을 수정한다.
   - 대조 후 사람 확인 절차를 확정합니다.
7. `docs/TESTING.md`의 현재 컴파일 확인 모드에 따라 컴파일 확인을 진행합니다.
   - `사람 확인` 모드에서는 사용자에게 Unity Editor 컴파일 확인을 요청합니다.
   - `Codex 확인` 모드에서는 Codex가 `build.ps1`로 컴파일 확인을 수행합니다.
8. 사용자에게 필요한 수동 작업 또는 사람 확인을 요청합니다.
9. 컴파일 확인과 필요한 사람 확인이 완료된 뒤 task 문서와 `docs/PLANS.md`에서 현재 단계 상태를 `완료` 로 갱신합니다.

## 진행 제약

- 하나의 task를 끝내기 전 다음 task로 넘어가지 않습니다.
- 저장소 파일 변경은 항상 적용 전에 변경 요약을 제시하고 사용자 승인 응답을 받은 뒤 진행합니다. task 문서 생성, 코드 구현, 기존 문서 규칙 변경도 여기에 포함됩니다.
- 컴파일 확인 주체는 `docs/TESTING.md`의 현재 컴파일 확인 모드를 따릅니다.
- 컴파일 오류가 있거나 사람의 컴파일 확인이 필요한 상태에서는 다음 단계로 넘어가지 않습니다.
- 사람 확인이 끝나기 전에는 다음 단계로 넘어가지 않습니다.
- 구현과 문서 갱신은 같은 작업 흐름 안에서 함께 반영합니다.

## 상태 관리 원칙

- 단계 상태는 `예정`, `진행중`, `완료`, `차단됨`만 사용합니다.
- 작업 로그에는 구현, 컴파일 확인, 사람 확인 결과를 가능한 한 남깁니다.
