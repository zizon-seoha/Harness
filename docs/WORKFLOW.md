# Workflow

## 문서 우선순위

1. 사용자 직접 요청
2. `CLAUDE.md`
3. 현재 진행 중인 task 문서
4. `docs/PLANS.md`
5. 나머지 프로젝트 문서

문서끼리 충돌하거나 현재 저장소 상태와 맞지 않으면, 구현 전에 불일치를 먼저 보고하고 수정 범위를 제안한다.

## 작업 흐름

1. 관련 task와 선행 산출물을 확인한다.
2. `docs/PLANS.md`에서 현재 단계를 `진행중`으로 갱신한다.
3. 관련 코드와 기존 구조를 읽고 재사용 가능 부분을 확인한다.
4. 필요한 코드 변경을 구현한다.
5. task 문서를 실제 구현 범위 기준으로 갱신한다 (`docs/TASK_WRITING.md`).
6. 변경 범위에 맞는 검증 명령을 실행한다 (`docs/TESTING.md` → `docs/STACK.md` 명령).
7. 사람 확인이 필요한 런타임·통합 항목이 있으면 사용자에게 요청한다.
8. 검증 명령 통과 + 필요한 사람 확인 완료 후 `docs/PLANS.md` 상태를 `완료`로 갱신한다.

## 진행 제약

- 하나의 task를 끝내기 전 다음 task로 넘어가지 않는다.
- 큰 변경(구조 변경, 광범위 리팩터)은 Plan Mode로 계획을 승인받고 진행한다.
- 변경 적용 권한은 `.claude/settings.json`의 allowlist를 전제로 한다.
- 검증 명령이 실패하거나 필요한 사람 확인이 끝나지 않은 상태에서는 다음 단계로 넘어가지 않는다.
- 구현과 문서 갱신은 같은 작업 흐름 안에서 함께 반영한다.

## 이력과 되돌리기

- 작업 이력은 git 커밋으로 남긴다. 별도 수동 작업 로그를 만들지 않는다.
- 되돌리기는 `/rewind` 또는 git을 쓴다.

## 상태 머신

단계 상태는 `예정`, `진행중`, `완료`, `차단됨`만 사용한다. 범례는 `docs/PLANS.md`.
