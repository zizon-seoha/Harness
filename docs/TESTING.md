# Testing

## 검증 원칙

각 task 구현 후 아래 순서로 검증합니다.

1. 관련 코드 수정 완료 여부 확인
2. 현재 컴파일 확인 모드에 맞는 정적 점검
3. Unity 및 C# 규칙 위반 여부 확인
4. 현재 task의 구현 범위와 완료 조건 일치 여부 확인
5. 진행 현황 문서 갱신

## 컴파일 확인 모드

현재 저장소의 컴파일 확인 모드는 `사람 확인`입니다.

- `사람 확인`: Codex는 컴파일을 직접 실행하지 않고, 사람이 Unity Editor에서 컴파일 결과를 확인합니다.
- `Codex 확인`: Codex가 `build.ps1`을 실행해 로컬 컴파일 확인을 수행합니다.

## 컴파일 확인

- 공통 기준은 아래와 같습니다.
  - 가능하면 현재 컴파일 확인 모드에서 허용하는 정적 점검을 먼저 수행합니다.
  - 최종 기준은 Unity Editor에서 컴파일 오류가 없다는 확인입니다.
  - 컴파일 오류가 있으면 다음 단계로 넘어가지 않습니다.

### 사람 확인 모드

- Codex는 기본적으로 컴파일을 직접 실행하지 않습니다.
- Codex는 코드 눈검토, 금지 API 검색, task 문서와 구현 범위 일치 확인, 필요한 Unity Editor 수동 작업 문서화까지만 수행합니다.
- Codex는 Unity 컴파일 실행, Unity 생성 파일 직접 보정, Editor import 결과를 미리 대체하는 파일 생성을 수행하지 않습니다.
- 사람은 Unity Editor에서 컴파일 결과를 확인하고 오류 없음 또는 오류 로그를 응답합니다.

### Codex 확인 모드

- PowerShell에서 로컬 컴파일 확인이 필요할 때는 직접 `dotnet build` 대신 프로젝트 루트의 `build.ps1`을 사용합니다.
- PowerShell에서 로컬 컴파일 확인 기본 명령은 `powershell -ExecutionPolicy Bypass -File build.ps1`입니다.

## 수동 검증 기준

- 자동 테스트가 없더라도 현재 단계 결과를 수동으로 검증할 수 있어야 합니다.
- 수동 검증 항목은 현재 구현된 범위만 대상으로 삼습니다.
- 아직 연결되지 않은 자동 흐름이나 다음 단계 전제를 검증 항목에 섞지 않습니다.
- 실패 시 어떤 현상을 기록해야 하는지 분명해야 합니다.
